# CLAUDE.md — Satisfaction Meter (AWS Marketing AI)

This file gives Claude the standing context it needs to work on this project across sessions. Read it at the start of every session. Keep it terse and current — if something changes, update this file rather than letting it drift.

## Project summary

**Satisfaction Meter** — a web-based emotion-driven marketing tool built as a **semester final project**. A user uploads a photo via a web interface; Amazon Rekognition detects the dominant facial emotion (happy, surprised, neutral, sad, etc.); a tailored marketing email is dispatched via SES based on the detected mood (e.g., happy → review request, sad → voucher/discount). Every result (photo reference, detected emotion, email sent) is persisted to a database and accessible via a GET API endpoint.

This runs on a **single free-tier AWS account**. Multiple end-users can submit photos (each provides their email); multiple developers share the same account via IAM users. Scope is intentionally narrow: web upload → emotion → email → stored result. No mobile app, no in-store kiosks.

**Note on production simulation**: certain design choices (e.g. 30-day image retention, SSE-S3 encryption, IAM least-privilege) are deliberately modelled on real production practice even though the project is short-lived. These exist to demonstrate good architectural judgement, not because they are operationally necessary for a semester timeline.

## Architectural stance

AWS-first. Prefer managed, serverless services over self-managed infrastructure unless there is a clear reason not to. The default stack:

- **Region**: `ap-southeast-1` (Singapore). All resources must be created here. Exception: ACM certificate for CloudFront must be in `us-east-1` (CloudFront requirement) — `arn:aws:acm:us-east-1:860550672813:certificate/<cert-id>` is the single approved cross-region resource. (Replace `<cert-id>` with the actual certificate ID from the ACM console.)
- **Image capture & upload**: Web frontend (Next.js 16 + React 19 + Tailwind 4, located in `frontend/`) uploads images to S3 via presigned URLs generated by an API Gateway + Lambda endpoint. No Amplify/Cognito needed — personal project, no auth layer required initially. Primary capture path is webcam: face-api.js runs client-side to show a green/red face-detection overlay and auto-snaps when a face is stable for 1.5s; the snapshot blob is then uploaded via the presigned URL flow. Manual file upload remains a fallback. face-api.js is overlay/UX only — Rekognition is the source of truth for emotion classification.
- **Frontend hosting**: Next.js is built as a fully static export (`output: "export"` in `frontend/next.config.ts`). The `out/` bundle is uploaded to a private S3 bucket and served via a CloudFront distribution with Origin Access Control (OAC). No public bucket access. Provisioned by `WebStack` (`lib/web-stack.ts`). Deploys via the `frontend-deploy.yml` GitHub Actions workflow on push to `main` when `frontend/**` changes — backend stacks are unaffected. Live at `https://satisfactionmeter.live` (apex domain) and `d1d3rdsk86mn4b.cloudfront.net` (CloudFront origin).
- **Image storage**: S3 with server-side encryption (SSE-S3 / AES-256 — AWS-managed keys, free tier compatible). Lifecycle rule: delete raw images after **30 days** (see retention decision in roadmap). Emotion features extracted before deletion.
- **Emotion detection**: Amazon Rekognition `DetectFaces` (returns emotion confidence scores). Triggered by S3 event → Lambda.
- **Routing & orchestration**: Lambda for stateless processing; Step Functions only when a workflow has more than ~3 sequential steps or needs retries with branching.
- **Identity**: Anonymous/session-based. The user provides their **email address** at upload time; this is the identity anchor used to route the outbound email. No loyalty ID or account system needed at this scale. Admin portal uses a single admin credential stored in SSM Parameter Store, validated by an API Gateway Lambda Authorizer.
- **Results storage**: DynamoDB — stores `{submissionId, email, s3Key, dominantEmotion, emotionScores, emailSentAt, templateUsed, timestamp}`. Accessible via a GET API endpoint.
- **Messaging**: **SES only** (transactional email). Domain `satisfactionmeter.live` acquired; sender address is `noreply@satisfactionmeter.live` (verified in SES). SES production access request submitted — pending AWS approval. Until approved, sending is limited to manually verified recipient addresses. Pinpoint, SNS, and SMS are out of scope. Emotion → template mapping: happy → review request, sad → voucher, neutral → general offer.
- **Analytics**: DynamoDB scan/query-based aggregation Lambdas behind `GET /analytics/*` endpoints. Phase 3A backend (emotions, campaigns, trends endpoints) is **done**. Phase 3B (admin portal frontend + Lambda Authorizer protecting routes) is in progress — admin dashboard frontend exists at `frontend/app/admin/dashboard/` with mock data in `frontend/lib/mockAnalytics.ts` pending swap to real calls (AWS-70). CloudWatch for operational metrics. No Kinesis/Firehose/Athena/QuickSight at this scale.
- **Security**: SSE-S3 for S3 encryption (free; KMS customer-managed keys cost $1/month — excluded). IAM least-privilege per Lambda role. **SSM Parameter Store** (Standard tier, free) for any credentials — Secrets Manager is excluded ($0.40/secret/month after trial). No VPC endpoints (hourly cost — not justified at this scale).
- **IaC**: AWS CDK (TypeScript). One stack per bounded context: `capture` (upload + S3), `inference` (Rekognition Lambda), `messaging` (SES send), `api` (GET results endpoint), `analytics` (analytics Lambdas + Lambda Authorizer + campaigns table), `web` (S3 + CloudFront for static frontend). Frontend lives in `frontend/` (Next.js); admin portal at `frontend/app/admin/`.
- **CI/CD**: GitHub Actions + OIDC. Two workflows: `deploy.yml` (CDK on backend changes — `frontend/**` is path-ignored) and `frontend-deploy.yml` (build static export → `aws s3 sync` → CloudFront invalidation, on `frontend/**` changes). OIDC means no AWS credentials stored in GitHub secrets — GitHub assumes an IAM role via federated identity.

## Working conventions

- **Stay AWS-native** unless there is a specific reason a third-party service is materially better. If you reach for something non-AWS, call it out explicitly and explain why.
- **Privacy by default**: customer images are sensitive. Never log image bytes or face attributes alongside PII. Drop raw images on a tight TTL once emotion features are extracted. Treat emotion scores themselves as personal data.
- **Region**: pick a single primary region and stick to it across the project. If unset in the roadmap, ask before introducing cross-region resources.
- **Free tier budget**: target $0 AWS spend. Flag any service or configuration that falls outside free tier. This is a semester project so 12-month free tier limits (Rekognition, API Gateway) are not a concern — usage will end well before expiry. The only real cost watch items are services with an unavoidable monthly fee regardless of usage: KMS CMKs ($1/month/key) and Secrets Manager ($0.40/secret/month) — both explicitly excluded. SES is ~$0 at expected volume.
- **Real-time SLO**: target end-to-end (image upload → email dispatched) under ~5 seconds for the web upload path.
- **Reliability**: async Lambdas (Rek Handler, SES Dispatcher) must have SQS dead-letter queues (DLQs) configured. SES bounce/complaint handling (SNS → Lambda webhook) is a real task on the roadmap — now that production access is in progress and real recipients will receive emails, bounce/complaint handling is required.

## How to work in this repo

1. **At session start**: read this file, then read `docs/roadmap.md` to see current phase, what's done, and what's next. Do not start work without checking the roadmap.
2. **Keep `docs/roadmap.md` current**: when you complete a milestone, decide a new approach, or learn something that changes plans, update the roadmap in the same change. The roadmap is the project's working memory between sessions — if it's stale, future sessions will redo work or miss context.
3. **Plan before building**: for any non-trivial change, sketch the approach (which AWS services, IAM boundaries, data flow) and confirm with the user before writing infrastructure code.
4. **Verify deployments**: after deploying CDK changes, confirm with `aws` CLI calls or console checks rather than assuming the stack is healthy.
5. **Document trade-offs in code**: when a decision isn't obvious (e.g., DynamoDB vs. Aurora for a table), leave a short comment with the reasoning so it survives.

## Out of scope (for now)

- On-device inference. Cloud Rekognition is the baseline.
- Custom ML models / on-device classification. Use Rekognition unless accuracy gaps force a SageMaker path — belongs in roadmap, not a side quest. (face-api.js is permitted for client-side overlay UX only, not for emotion classification.)
- Multi-tenant / account-level isolation.
- Mobile app or in-store kiosk clients. Web upload only.
- SMS channel (SNS/Pinpoint). Email via SES only.
- Kinesis, Firehose, Athena, QuickSight. Out of scope at personal-project scale.

## Consent & privacy stance

Personal project; no regulated consent flow required. If displayed publicly (e.g., a demo kiosk), a brief notice ("Photos are processed for emotion detection and deleted within 30 days") is sufficient. Emotion scores are still treated as personal data in code — never log alongside PII, never store raw face bytes beyond the retention window.
